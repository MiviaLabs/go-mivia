# Mivia v0.2.3 Release Notes

Date: 2026-06-04

## Highlights

- Added governed automation runner GitOps so supervised external `mivia-automation-runner` attempts can commit bounded task changes, optionally push task branches, and optionally create or reuse draft GitHub PRs through fixed `git` and `gh` templates.
- Added dedicated-worktree execution support and worktree schema hardening for automation workers.
- Stabilized runner Codex execution, task-status handling, automatic review-gate workflow execution, and external-runner verifier-required state transitions.
- Added the Compose UID helper and devcontainer/Compose runner examples so automation commits, refs, and worktree metadata stay owned by the host checkout user.
- Hardened global git config handling and MCP diagnostics for automation and workspace flows.

## Safety And Governance

- Automation remains an executor over existing Work Plans and ready Work Tasks. It does not complete tasks, self-review, run verifier gates, promote knowledge, or store raw prompts, completions, source dumps, raw stderr, provider payloads, secrets, roots, external URLs, or PII.
- GitOps is disabled by default and fails closed unless a task supplies safe scoped affected paths. It does not stage whole worktrees as a fallback.
- Push and draft PR creation require configured SSH and GitHub auth references. Docs and config examples must not contain token literals, key material, raw command output, roots, or stderr.
- Direct MCP workspace tools still do not expose git commit, push, checkout, reset, branch, merge, rebase, stash, clean, or restore operations.

## Verification Scope

- Release verification should include targeted automation, GitOps, worktree, workflow, config, MCP compatibility, and git diff hygiene checks.
- Full `go test ./...` should be run by release verification before tagging.
